FIREFLY III ON DOCKER — TROUBLESHOOTING
CNM260  | github.com/GAustin355/CNM260

Common errors, their causes, and how to fix them. Sorted by what shows up
most often during a first deployment.

ERROR: "port is already allocated" / "Bind for 0.0.0.0:8080 failed"

CAUSE: Another program on your machine is using port 8080.

FIX: Open docker-compose.yml. Find:

    ports:
      - "8080:8080"

Change the LEFT number to anything free (9090, 8888, etc.):

    ports:
      - "9090:8080"

Save, then run:

    docker compose up -d

Open http://localhost:9090 in your browser instead.

ERROR: "App key is not set"

CAUSE: Either you forgot to fill in APP_KEY=, or your APP_KEY contains one
of these forbidden characters: # < > = &

FIX: Regenerate the APP_KEY using the command from the README and paste the
new value into .env. Then:

    docker compose down
    docker compose up -d

ERROR: "SQLSTATE[HY000] [2002] Connection refused" in app logs

CAUSE: The app is trying to talk to the database before MariaDB is fully
ready. This usually fixes itself within 30 seconds.

FIX: Wait 30 more seconds, then check:

    docker compose logs -f app

If it persists for over a minute, check the database logs:

    docker compose logs db

Look for [ERROR] lines. The most common root cause is the next item below.

ERROR: "Access denied for user 'firefly'@'%'" or similar auth failure

CAUSE: MYSQL_PASSWORD in .db.env does NOT match DB_PASSWORD in .env.

THE CATCH: MariaDB sets the password the FIRST time it starts and writes it
to the database volume. If you ran docker compose up once, then changed the
password in .db.env, the new password is NOT actually applied — the volume
already has the old one.

FIX: You have to destroy the database volume and start over:

    docker compose down -v   # WARNING: deletes all data!

Now fix the passwords so they match exactly in BOTH files. Then:

    docker compose up -d --pull=always

ERROR: 500 Internal Server Error - "getaddrinfo for db failed"

CAUSE: The app container can't find the db container on the Docker network.
Usually means the db container failed to start, OR Docker Desktop stopped
running.

FIX:

1. Check if Docker Desktop is running (whale icon steady in system tray)

2. Check container status:

       docker compose ps

   If db shows "Exited", check why:

       docker compose logs db

3. Most common fix - reset the volumes:

       docker compose down -v
       docker compose up -d --pull=always

ERROR: Login page loads but credentials don't work after a reset

CAUSE: You regenerated APP_KEY after creating your first user. The APP_KEY
encrypts session and password data — change it after the fact and existing
sessions/passwords break.

FIX (fresh install, nothing to lose):

    docker compose down -v
    docker compose up -d --pull=always

Re-register the first user.

FIX (data you want to keep):
    Restore the original APP_KEY from your backup of .env. If you didn't
    keep one, the data is recoverable but it requires database surgery —
    see https://docs.firefly-iii.org/references/faq/install/

ERROR: "docker compose: command not found" or "docker: command not found"

CAUSE 1: Docker Desktop is installed but not running.

FIX: Launch Docker Desktop. Wait for the whale icon to be steady (not
animating). Try the command again.

CAUSE 2: You have the older standalone docker-compose (with a hyphen)
instead of the modern Docker Compose plugin.

FIX: Update Docker Desktop / Docker Engine to a current version. Or, run
the legacy command instead:

    docker-compose up -d --pull=always

ERROR: "Cannot connect to the Docker daemon" (Windows or Mac)

CAUSE: Docker Desktop isn't running.

FIX:

1. Open Docker Desktop from the Start menu (Windows) or Applications (Mac)

2. Wait 30-60 seconds for the whale icon in your system tray / menu bar
   to stop animating

3. Verify it's up:

       docker version

   You should see both a "Client" and a "Server" block with version numbers.

ERROR: "permission denied while trying to connect to the Docker daemon"
       (Linux only)

CAUSE: Your user is not in the `docker` group.

FIX:

    sudo usermod -aG docker $USER

Then LOG OUT AND BACK IN (the group change doesn't apply to existing
sessions). Try the command again.

ERROR: Containers start but http://localhost:8080 doesn't respond

CAUSE 1: The app is still doing first-run migrations (60+ seconds).

FIX: Wait. Run `docker compose logs -f app` and look for the migration
output to finish.

CAUSE 2: A firewall is blocking localhost:8080.

FIX: Temporarily disable your firewall and try again. If it works, add an
exception for port 8080.

CAUSE 3: You changed the host port in docker-compose.yml but are still
trying to access 8080.

FIX: Use the new port. If your compose file says "9090:8080", visit
http://localhost:9090 instead.

ERROR: Everything works on Linux/Mac but not on Windows

CAUSE: Docker Desktop is running on Hyper-V mode instead of WSL 2.

FIX: Open Docker Desktop -> Settings -> General. Make sure "Use the WSL 2
based engine" is checked. Apply, restart Docker Desktop.

ERROR: docker compose up succeeds but `docker compose ps` shows "exited"

CAUSE: Something inside the container crashed during startup.

FIX: Check the logs to see why:

    docker compose logs app
    docker compose logs db

The error message will be in there. Most common reasons:
  - APP_KEY missing or contains forbidden characters
  - DB password mismatch (see above)
  - MariaDB volume corrupted from a previous failed install

If the database container won't stay up and you've already verified
passwords match, nuke it:

    docker compose down -v
    docker compose up -d --pull=always

RESET BUTTON — when you want to start completely over

WARNING: this destroys all transaction data and uploaded files.

    docker compose down -v
    docker system prune -f
    cp .env.example .env
    cp .db.env.example .db.env
    # re-edit .env and .db.env with new APP_KEY and matching passwords
    docker compose up -d --pull=always
